This style guide helps you write clear, consistent Runpod documentation. Use these conventions for terminology, formatting, and structure so readers can find what they need quickly.

## Terminology

Runpod documentation uses specific capitalization rules for product names and technical terms.

### Proper nouns

Always capitalize these Runpod products and features:

| Term | Usage |
|------|-------|
| Runpod | The company and platform name |
| Pods | Dedicated containerized computing instances |
| Serverless | Pay-per-second auto-scaling computing |
| Flash | Python SDK for deploying functions to cloud GPUs |
| Hub | AI repository marketplace |
| Instant Clusters | Multi-node distributed computing |
| Secure Cloud | Enterprise-grade cloud partner program |
| Community Cloud | Community-hosted GPU resources |
| Tetra | Runpod's workload orchestration system |
| Public Endpoint | Pre-deployed models accessible via REST APIs |

### Generic terms

Use lowercase for general technical terms:

- endpoint
- worker
- cluster
- template
- handler
- fine-tune
- network volume
- data center
- repo

### Examples

**Correct:**

- "Deploy a Pod using the Runpod console."
- "Create a Serverless endpoint for your handler."
- "Instant Clusters provide high-speed networking."

**Incorrect:**

- "Deploy a pod using the RunPod console."
- "Create a serverless Endpoint for your Handler."

## Writing style

### Voice and tone

Write in a clear, direct style that helps users accomplish their goals.

- **Use second person.** Address readers as "you" rather than using "we" or "one."
- **Prefer active voice.** Write "Deploy a Pod" instead of "A Pod is deployed."
- **Use present tense.** Write "The command creates a new Pod" instead of "The command will create a new Pod."
- **Keep sentences focused.** Each sentence should convey one idea.

### First person

Avoid first person pronouns ("I", "we", "our") in most cases. Instead of "We recommend that you configure," write "You should configure" or "Configure."

This guideline is flexible for tutorials and conversational content where a more personal tone helps readers.

### Paragraphs and lists

Prefer paragraphs over bullet points for explanatory content. Paragraphs provide better context and flow for readers working through documentation.

Use bullet points when listing discrete, parallel items, such as:

- A set of requirements.
- A collection of options.
- A series of related resources.

When using bullet points, end each item with a period.

## Headings

### Sentence case

Always use sentence case for headings and titles. Capitalize only the first word and proper nouns.

**Correct:**
- "Create a serverless endpoint"
- "Connect to your Pod"
- "Configure Instant Clusters"

**Incorrect:**
- "Create a Serverless Endpoint"
- "Connect To Your Pod"

### Hierarchy

Structure pages using this heading hierarchy:

- **H1 (`#`):** Optional page title (if omitted, the frontmatter `title` displays).
- **H2 (`##`):** Major sections.
- **H3 (`###`):** Subsections within major sections.

Avoid skipping heading levels. Don't jump from H2 to H4.

## Document structure

### Frontmatter

Every page requires frontmatter with these fields:

```yaml
---
title: "Page title"
description: "One to two sentence summary for SEO and page previews."
---
```

Optional frontmatter fields:

- `sidebarTitle`: Shorter title for the navigation sidebar.
- `icon`: Icon name from the FontAwesome library.
- `iconType`: Icon style ("solid", "regular", "brands").

### Page layout

Most documentation pages follow this structure:

1. **Introduction**: A brief paragraph explaining the page's purpose.
2. **Requirements** (if applicable): What users need before following the instructions.
3. **Main content**: The core information, organized with H2 headings.
4. **Next steps** (optional): Links to related pages using CardGroup.

### Requirements section

Include a "Requirements" section (not "Prerequisites") below the introduction when users need specific setup before proceeding.

```markdown
## Requirements

- A Runpod account with credits
- Docker installed locally
- Basic familiarity with Python
```

## Tutorial format

Tutorials guide users through a complete workflow. Structure tutorials with these conventions.

### Opening section

Start tutorials with a brief description of what users will accomplish, followed by a "Requirements" section listing what they need.

### Numbered steps

Use numbered H2 headings for each major step:

```markdown
## Step 1: Create a template

Navigate to the Templates page and click **Create Template**.

## Step 2: Configure the environment

Set the following environment variables...

## Step 3: Deploy the endpoint

Click **Deploy** and wait for the endpoint to become active.
```

Steps should represent logical milestones, not individual clicks. Use numbered lists within steps for detailed sub-procedures.

### Clean up section

Include a final step reminding users to delete resources they no longer need to avoid charges.

## Code examples

### Language identifiers

Always specify the language in code blocks for syntax highlighting:

````markdown
```python
import runpod
```
````

Common identifiers: `python`, `bash`, `sh`, `json`, `yaml`, `javascript`, `typescript`, `curl`.

### File titles

Include a file title when the code represents a specific file:

````markdown
```python handler.py
def handler(job):
    return {"output": "Hello!"}
```
````

### Placeholders

Use `ALL_CAPS` with underscores for placeholder values that users must replace:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" \
  https://api.runpod.ai/v2/ENDPOINT_ID/run
```

Common placeholders:
- `YOUR_API_KEY`
- `RUNPOD_API_KEY`
- `ENDPOINT_ID`
- `POD_ID`
- `YOUR_POD_ID`
- `IMAGE_NAME`

### Code explanations

When explaining code, follow this sequence:

1. A brief introduction explaining what the code accomplishes.
2. The code sample.
3. An explanation of key parts.

Example:

````markdown
Create a handler function that processes text inputs:

```python handler.py
import runpod

def handler(job):
    text = job["input"]["text"]
    result = text.upper()
    return {"output": result}

runpod.serverless.start({"handler": handler})
```

The `handler` function extracts the `text` field from the input and returns it converted to uppercase. The `runpod.serverless.start()` call registers the handler with the Runpod runtime.
````

### Command variants

When showing alternative ways to run a command, use inline comments rather than separate blocks:

```bash
flash login

# If using uv:
uv run flash login
```

## Formatting

### Inline code

Use backticks for:

- Code snippets: `runpod.serverless.start()`
- File paths: `/workspace/handler.py`
- Environment variables: `RUNPOD_API_KEY`
- API endpoints: `/v2/endpoint_id/run`
- CLI commands: `runpodctl pod list`
- Parameter names: `gpuCount`

### Bold and emphasis

Use **bold** for:

- UI elements users interact with: Click **Deploy**.
- Key terms being introduced: A **worker** processes incoming requests.

Use *italics* sparingly for emphasis.

### Links

For internal links, use relative paths without file extensions:

```markdown
See the [Pods overview](/pods/overview) for more information.
```

For external links, use full URLs and descriptive link text:

```markdown
Refer to the [PyTorch distributed training documentation](https://pytorch.org/tutorials/beginner/dist_overview.html).
```

Avoid generic link text like "click here" or "this page."

## Mintlify components

Runpod documentation uses Mintlify's built-in components for enhanced formatting.

### Callouts

Use callouts for information that deserves special attention:

```markdown
<Note>
Additional helpful context that supplements the main content.
</Note>

<Tip>
A suggestion that helps users work more efficiently.
</Tip>

<Warning>
Important information about potential issues or gotchas.
</Warning>

<Danger>
Critical warning about data loss or irreversible actions.
</Danger>
```

### Tabs

Use tabs to present multiple interfaces or options:

```markdown
<Tabs>
<Tab title="Web">
Instructions for the web console...
</Tab>
<Tab title="CLI">
Instructions using the command line...
</Tab>
<Tab title="REST API">
Instructions using the REST API...
</Tab>
</Tabs>
```

### Steps component

Use the Steps component for procedural instructions within a section:

```markdown
<Steps>
<Step title="Create the file">
Create a new file called `handler.py` in your project directory.
</Step>
<Step title="Add the handler code">
Paste the following code into the file...
</Step>
</Steps>
```

### Cards

Use CardGroup for "Next steps" and "Learn more" sections instead of bullet lists:

```markdown
<CardGroup cols={2}>
  <Card title="Deploy your first Pod" href="/pods/manage-pods" icon="rocket" horizontal>
    Learn how to create and manage Pods.
  </Card>
  <Card title="Serverless quickstart" href="/serverless/quickstart" icon="bolt" horizontal>
    Build your first serverless endpoint.
  </Card>
</CardGroup>
```

### Frames

Wrap images in Frame components for consistent styling:

```markdown
<Frame>
  <img src="/images/example-screenshot.png" alt="Description of the image" />
</Frame>
```

### Accordions

Use accordions for supplementary information that not all readers need:

```markdown
<Accordion title="Environment variables reference">
| Variable | Description |
|----------|-------------|
| `MASTER_ADDR` | Address of the primary node |
| `NUM_NODES` | Number of nodes in the cluster |
</Accordion>
```

## Images

### File naming

Use descriptive, lowercase filenames with hyphens:

- `pods-deployment-flow.png`
- `serverless-architecture-diagram.png`
- `instant-clusters-rank-diagram.png`

### Alt text

Include descriptive alt text for accessibility:

```markdown
<Frame>
  <img src="/images/pod-creation-form.png" alt="Pod creation form showing GPU selection and configuration options" />
</Frame>
```

### Placement

Place images near the text they illustrate. For step-by-step instructions, include screenshots after describing what users should see.

## Tooltips

Tooltips provide hover definitions for technical terms. Import and use them for Runpod-specific concepts that readers might not know.

```markdown
import { PodTooltip, TemplateTooltip } from "/snippets/tooltips.jsx";

Deploy your first GPU <PodTooltip /> using a <TemplateTooltip />.
```

Most tooltips have singular and plural variants: `PodTooltip` and `PodsTooltip`.
