Remote functions are the core building blocks of Flash. The `@remote` decorator marks Python functions for execution on Runpod's Serverless infrastructure, handling resource provisioning, dependency installation, and data transfer automatically.

## How remote functions work

A remote function is just a Python function that's been marked with the `@remote` decorator. For example:

```python
@remote(resource_config=config, dependencies=["torch"])
def run_inference(data):
    import torch
    # Your inference code here
    return result
```

When you call a remote function from a local Python script or [Flash app](/flash/apps/overview), the function code is sent to a Runpod worker. The worker executes the function code and returns the result to your local environment.

## Resource configuration

Every remote function requires a resource configuration specifying compute resources. Flash provides six configuration types:

- **Queue-based** (`LiveServerless`, `CpuLiveServerless`) - Batch processing with `/run` or `/runsync` routes
- **Load balancing** (`LiveLoadBalancer`, `CpuLiveLoadBalancer`) - HTTP APIs with custom routes
- **Custom Docker** (`ServerlessEndpoint`, `CpuServerlessEndpoint`) - Custom Docker images

**Quick example:**

```python
from runpod_flash import LiveServerless, GpuType, remote

config = LiveServerless(
    name="ml-inference",
    gpus=[GpuType.NVIDIA_A100_80GB_PCIe],
    workersMax=5
)

@remote(resource_config=config, dependencies=["torch"])
async def process(data):
    import torch
    # Your code here
    return result
```

See [Configuration overview](/flash/configuration/overview) to choose the right configuration type for your workload, or jump directly to [Queue-based](/flash/configuration/queue-based), [Load balancing](/flash/configuration/load-balancing), or [Custom Docker](/flash/configuration/custom-docker-images) guides.

## Dependency management

Specify Python packages in the `dependencies` parameter of the `@remote` decorator. Flash installs these packages on the remote worker before executing your function.

```python
@remote(
    resource_config=config,
    dependencies=["transformers==4.36.0", "torch", "pillow"]
)
def generate_image(prompt):
    from transformers import pipeline
    import torch
    from PIL import Image
    # Your code here
```

<Tip>
Some packages (like PyTorch) are pre-installed on GPU workers, but including them in dependencies ensures the correct version is available.
</Tip>


### Import packages inside the function body

You must import packages **inside the decorated function body,** not at the top of your file. This will ensure the imports happen on the remote worker, not in your local environment.


**Correct:** imports inside the function.
```python
@remote(resource_config=config, dependencies=["numpy"])
def compute(data):
    import numpy as np  # Import here
    return np.sum(data)
```
**Incorrect:** imports at top of file won't work.

```python
import numpy as np  # This import happens locally, not on the worker

@remote(resource_config=config, dependencies=["numpy"])
def compute(data):
    return np.sum(data)  # numpy not available on the remote worker
```

### Version pinning

You can pin specific versions using standard pip syntax:

```python
dependencies=["transformers==4.36.0", "torch>=2.0.0"]
```

## Parallel execution

Remote functions can be written as `def` or `async def`. In both cases, calling them returns an awaitable that must be used with `await`. Use Python's `asyncio` to run multiple functions in parallel:

```python
import asyncio

async def main():
    # Run three functions in parallel
    results = await asyncio.gather(
        process_item(item1),
        process_item(item2),
        process_item(item3)
    )
    return results
```

This is particularly useful for:

- Batch processing multiple inputs.
- Running different models on the same data.
- Parallelizing independent pipeline stages.

### Example: Parallel batch processing

```python
import asyncio
from runpod_flash import remote, LiveServerless, GpuGroup

config = LiveServerless(
    name="batch-processor",
    gpus=[GpuGroup.ADA_24],
    workersMax=5  # Allow up to 5 parallel workers
)

@remote(resource_config=config, dependencies=["torch"])
def process_batch(batch_id, data):
    import torch
    # Process batch
    return {"batch_id": batch_id, "result": len(data)}

async def main():
    batches = [
        (1, [1, 2, 3]),
        (2, [4, 5, 6]),
        (3, [7, 8, 9])
    ]
    
    # Process all batches in parallel
    results = await asyncio.gather(*[
        process_batch(batch_id, data) 
        for batch_id, data in batches
    ])
    
    print(results)

if __name__ == "__main__":
    asyncio.run(main())
```

## Using persistent storage

Attach network volumes to share models, datasets, and files across workers using the `networkVolumeId` parameter. Volumes mount at `/runpod-volume/` and persist across worker restarts. See the [storage guide](/flash/configuration/storage) for setup instructions, access patterns, and best practices.

## Environment variables

Pass environment variables to remote functions using the `env` parameter in your configuration:

```python
config = LiveServerless(
    name="api-worker",
    env={
        "HF_TOKEN": "your_huggingface_token",
        "MODEL_ID": "gpt2",
        "API_KEY": "your_api_key"
    }
)
```

### Accessing environment variables in remote functions

Access environment variables inside your remote functions using Python's `os.getenv()`:

```python
@remote(resource_config=config, dependencies=["transformers"])
def load_model():
    import os
    from transformers import AutoModel

    # Access environment variables set in the configuration
    hf_token = os.getenv("HF_TOKEN")
    model_id = os.getenv("MODEL_ID")

    # Use them in your function logic
    model = AutoModel.from_pretrained(
        model_id,
        use_auth_token=hf_token
    )

    return {"model_loaded": model_id}
```

You can also provide default values if the environment variable is not set:

```python
@remote(resource_config=config)
def process_data(data):
    import os

    # Get environment variable with a default value
    log_level = os.getenv("LOG_LEVEL", "INFO")
    max_retries = int(os.getenv("MAX_RETRIES", "3"))

    # Your processing logic here
    return {"processed": True}
```

<Note>
You can change environment variables without recreating your endpoint. This makes it easy to rotate API keys or allow different developers to use their own .env files.
</Note>

## Decorator parameters

The `@remote` decorator accepts the following parameters:

| Parameter | Type | Description | Default |
|-----------|------|-------------|---------|
| `resource_config` | `ServerlessResource` | Configuration object specifying compute resources (required) | - |
| `dependencies` | `list[str]` | Python packages to install (pip syntax) | `None` |
| `system_dependencies` | `list[str]` | System packages to install (apt packages) | `None` |
| `accelerate_downloads` | `bool` | Enable download acceleration for dependencies and models | `True` |
| `local` | `bool` | Execute function locally instead of on remote workers | `False` |
| `method` | `str` | HTTP method for load-balanced endpoints | `None` |
| `path` | `str` | HTTP path for load-balanced endpoints | `None` |

### System dependencies

Use `system_dependencies` to install system-level packages (via apt) before your function runs:

```python
@remote(
    resource_config=config,
    dependencies=["opencv-python"],
    system_dependencies=["libgl1-mesa-glx", "libglib2.0-0"]  # Required by OpenCV
)
async def process_image(image_data):
    import cv2
    # OpenCV processing
    return {"processed": True}
```

### Download acceleration

The `accelerate_downloads` parameter (default: `True`) enables faster downloads for dependencies, models, and large files. Disable it if you encounter compatibility issues:

```python
@remote(
    resource_config=config,
    dependencies=["transformers"],
    accelerate_downloads=False  # Disable if needed
)
async def load_model():
    from transformers import AutoModel
    return AutoModel.from_pretrained("bert-base-uncased")
```

### Local execution mode

Use `local=True` to run a function locally instead of on remote workers. This is useful for testing and development without incurring cloud costs:

```python
@remote(
    resource_config=config,
    dependencies=["numpy"],
    local=True  # Execute locally for testing
)
async def test_function(data):
    import numpy as np
    return {"sum": np.sum(data)}
```

<Warning>
When using `local=True`, you must have all dependencies installed in your local environment. Flash does not install them for you.
</Warning>

### HTTP routing (load balancers only)

For load-balanced endpoints, `method` and `path` are required to define HTTP routes:

```python
from runpod_flash import LiveLoadBalancer, remote

config = LiveLoadBalancer(name="api-server")

@remote(resource_config=config, method="POST", path="/api/process")
async def process_data(data: dict) -> dict:
    return {"result": "processed"}

@remote(resource_config=config, method="GET", path="/health")
async def health_check() -> dict:
    return {"status": "healthy"}
```

See [Load balancing configurations](/flash/configuration/load-balancing) for complete details.
