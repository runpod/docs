Queue-based configurations create Serverless endpoints that process jobs asynchronously through a queue. Jobs are submitted via `/run` (async) or `/runsync` (sync with timeout) routes.

## When to use queue-based configurations

Choose queue-based configurations when you need:

- **Batch processing**: Process multiple items over time.
- **Asynchronous workloads**: Jobs that can wait in a queue.
- **GPU-intensive tasks**: Model training, image generation, video processing.
- **Job retries and tracking**: Automatic retry handling and job status monitoring.
- **Long-running tasks**: Operations that may take minutes to complete.

## LiveServerless

`LiveServerless` is the primary configuration for GPU-accelerated batch processing. It supports full remote code execution, allowing you to run arbitrary Python functions on Runpod's GPU infrastructure.

### Basic example

```python
from runpod_flash import LiveServerless, GpuGroup, remote

config = LiveServerless(
    name="ml-inference",
    gpus=[GpuGroup.AMPERE_80],  # A100 80GB
    workersMax=5,
    idleTimeout=10
)

@remote(resource_config=config, dependencies=["torch"])
async def run_inference(data: dict) -> dict:
    import torch
    # GPU processing logic
    return {"result": "processed"}
```

### Configuration

Key parameters: `name` (required), `gpus`, `workersMax`, `idleTimeout`, `env`.

See [Configuration parameters](/flash/configuration/parameters) for the complete parameter reference.

### GPU selection

The `gpus` parameter accepts specific GPU types for precise hardware selection:

```python
from runpod_flash import LiveServerless, GpuType

config = LiveServerless(
    name="ml-inference",
    gpus=[GpuType.NVIDIA_A100_80GB_PCIe]
)
```

See [GPU types](/flash/configuration/gpu-types) for all available GPU types, pools, and advanced selection patterns.

### Multi-GPU workers

Request multiple GPUs per worker using `gpuCount`:

```python
config = LiveServerless(
    name="multi-gpu-training",
    gpus=[GpuGroup.AMPERE_80],
    gpuCount=4,  # Each worker gets 4 GPUs
    workersMax=2  # Maximum 2 workers (8 GPUs total)
)
```

### Complete example

```python
from runpod_flash import LiveServerless, GpuGroup, PodTemplate, remote

config = LiveServerless(
    name="gpu-inference",
    gpus=[GpuGroup.AMPERE_80, GpuGroup.AMPERE_48],
    gpuCount=1,
    workersMin=0,
    workersMax=5,
    idleTimeout=10,
    executionTimeoutMs=300000,  # 5 minutes
    networkVolumeId="vol_abc123",
    template=PodTemplate(containerDiskInGb=100),
    env={"MODEL_ID": "llama-7b", "HF_TOKEN": "your_token"}
)

@remote(resource_config=config, dependencies=["torch", "transformers"])
async def generate_text(prompt: str) -> dict:
    import torch
    from transformers import AutoModelForCausalLM, AutoTokenizer

    # Load model from network volume
    model_path = "/runpod-volume/models/llama-7b"
    model = AutoModelForCausalLM.from_pretrained(model_path)
    tokenizer = AutoTokenizer.from_pretrained(model_path)

    # Generate text
    inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
    outputs = model.generate(**inputs, max_length=100)
    text = tokenizer.decode(outputs[0])

    return {"generated_text": text}
```

## CpuLiveServerless

`CpuLiveServerless` is optimized for CPU-only batch processing. Use it for workloads that don't require GPU acceleration.

### Basic example

```python
from runpod_flash import CpuLiveServerless, CpuInstanceType, remote

config = CpuLiveServerless(
    name="data-processor",
    instanceIds=[CpuInstanceType.CPU5C_4_8],  # 4 vCPU, 8GB RAM
    workersMax=3,
    idleTimeout=5
)

@remote(resource_config=config, dependencies=["pandas"])
async def process_csv(file_url: str) -> dict:
    import pandas as pd
    df = pd.read_csv(file_url)
    return {"rows": len(df), "columns": list(df.columns)}
```

### Configuration

Key parameters: `name` (required), `instanceIds` (required), `workersMax`, `idleTimeout`, `env`.

See [Configuration parameters](/flash/configuration/parameters) for the complete parameter reference.

### CPU instance selection

Specify CPU instance types using `instanceIds`:

```python
from runpod_flash import CpuLiveServerless, CpuInstanceType

# Single instance type
config = CpuLiveServerless(
    name="cpu-worker",
    instanceIds=[CpuInstanceType.CPU5C_4_8]  # 4 vCPU, 8GB RAM
)

# Multiple instance types (fallback strategy)
config = CpuLiveServerless(
    name="flexible-cpu",
    instanceIds=[
        CpuInstanceType.CPU5C_8_16,  # Try 8 vCPU, 16GB first
        CpuInstanceType.CPU5C_4_8,   # Fall back to 4 vCPU, 8GB
        CpuInstanceType.CPU3G_4_16   # Final fallback
    ]
)
```

See [CPU types reference](/flash/configuration/cpu-types) for all available CPU instance types.

### Complete example

```python
from runpod_flash import CpuLiveServerless, CpuInstanceType, PodTemplate, remote

config = CpuLiveServerless(
    name="batch-processor",
    instanceIds=[CpuInstanceType.CPU5C_8_16],
    workersMin=1,
    workersMax=10,
    idleTimeout=15,
    executionTimeoutMs=600000,  # 10 minutes
    template=PodTemplate(containerDiskInGb=50),
    env={"API_KEY": "your_api_key"}
)

@remote(resource_config=config, dependencies=["requests", "pandas"])
async def process_batch(urls: list) -> dict:
    import requests
    import pandas as pd

    results = []
    for url in urls:
        response = requests.get(url)
        results.append(response.json())

    df = pd.DataFrame(results)
    return {
        "processed": len(results),
        "summary": df.describe().to_dict()
    }
```

<Note>
Use `CpuLiveServerless` for CPU workloads. `LiveServerless` is designed for GPU workloads and uses a GPU-optimized Docker image that is not appropriate for CPU-only deployments.
</Note>

## Calling queue-based endpoints

After deploying with `flash deploy`, call your endpoints using `/run` (async) or `/runsync` (sync) routes. See [Send API requests](/flash/apps/requests#queue-based-endpoints) for complete details on request formats, authentication, and status checking.

## Handling GPU unavailability

If no GPUs in your list are available, jobs will stay in queue:

```text
Initial job status: IN_QUEUE
[Job waits for capacity]
```

**Solutions:**

1. **Check console**: View [GPU availability](https://www.runpod.io/console/serverless) in the Runpod console
2. **Add more GPU types**: Expand your `gpus` list to include fallback options
3. **Use GpuGroup.ANY**: Switch to `[GpuGroup.ANY]` for maximum availability
4. **Contact support**: If capacity is consistently unavailable, contact [Runpod support](https://www.runpod.io/contact)

## Worker scaling

Queue-based endpoints scale workers based on queue depth. Set `workersMin=0` for cost optimization (scale to zero when idle) or `workersMin ≥ 1` for low-latency production workloads. Jobs can wait in queue during scaling, making queue-based endpoints ideal for batch processing where immediate response isn't critical.

```python
# Cost-optimized: Scale to zero when idle
config = LiveServerless(
    name="batch-job",
    workersMin=0,
    workersMax=10,
    idleTimeout=5
)
```

See [Configuration parameters](/flash/configuration/parameters) for detailed scaling parameter behavior.

## Persistent storage

For persistent storage, set `networkVolumeId` to your volume ID. Network volumes mount at `/runpod-volume/` and persist across workers. See the [storage guide](/flash/configuration/storage) for setup instructions and best practices.
