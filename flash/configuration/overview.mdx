Flash provides six resource configuration types for different workload patterns. This guide helps you choose the right one.

## Overview

| Configuration | When to use | Endpoint pattern |
|---------------|-------------|------------------|
| `LiveServerless` | GPU batch processing | Queue-based (`/run`, `/runsync`) |
| `CpuLiveServerless` | CPU batch processing | Queue-based (`/run`, `/runsync`) |
| `LiveLoadBalancer` | GPU-powered HTTP APIs | Custom routes (`POST /inference`) |
| `CpuLiveLoadBalancer` | CPU-powered HTTP APIs | Custom routes (`GET /status`) |
| `ServerlessEndpoint` | Pre-built GPU Docker images | Queue-based (`/run`, `/runsync`) |
| `CpuServerlessEndpoint` | Pre-built CPU Docker images | Queue-based (`/run`, `/runsync`) |

## Queue-based endpoints

Queue-based configurations process jobs asynchronously through a queue. Jobs are submitted via `/run` (async) or `/runsync` (sync with timeout). Use these for batch processing, long-running tasks, or workloads that don't need immediate responses.

### LiveServerless

GPU-accelerated batch processing with remote code execution.

```python
from runpod_flash import LiveServerless, GpuGroup, remote

config = LiveServerless(
    name="gpu-worker",
    gpus=[GpuGroup.AMPERE_80],  # A100 80GB
    workersMax=5,
    idleTimeout=10
)

@remote(resource_config=config, dependencies=["torch"])
async def process_data(data: dict) -> dict:
    import torch
    # Your GPU code here
    return {"result": "processed"}
```

### CpuLiveServerless

CPU-only batch processing with remote code execution.

```python
from runpod_flash import CpuLiveServerless, CpuInstanceType, remote

config = CpuLiveServerless(
    name="cpu-worker",
    instanceIds=[CpuInstanceType.CPU5C_4_8],  # 4 vCPU, 8GB RAM
    workersMax=3,
    idleTimeout=5
)

@remote(resource_config=config, dependencies=["pandas"])
async def process_csv(file_url: str) -> dict:
    import pandas as pd
    # Your CPU code here
    return {"rows": 100}
```

## Load balancing endpoints

Load balancing configurations handle direct HTTP requests with custom routes. Use these for REST APIs, webhooks, or any workload requiring synchronous request/response patterns. Multiple `@remote` functions can share the same configuration, each defining its own HTTP route.

### LiveLoadBalancer

GPU-accelerated HTTP APIs with custom routes.

```python
from runpod_flash import LiveLoadBalancer, GpuGroup, remote

config = LiveLoadBalancer(
    name="gpu-api",
    gpus=[GpuGroup.ADA_24],  # RTX 4090
    workersMin=1,
    workersMax=5,
    idleTimeout=10
)

@remote(resource_config=config, method="POST", path="/inference")
async def run_inference(data: dict) -> dict:
    import torch
    # Your GPU inference code here
    return {"prediction": "result"}

@remote(resource_config=config, method="GET", path="/health")
async def health_check() -> dict:
    return {"status": "healthy"}
```

### CpuLiveLoadBalancer

CPU-only HTTP APIs with custom routes.

```python
from runpod_flash import CpuLiveLoadBalancer, remote

config = CpuLiveLoadBalancer(
    name="cpu-api",
    workersMin=1,
    workersMax=10,
    idleTimeout=10
)

@remote(resource_config=config, method="POST", path="/process")
async def process_data(data: dict) -> dict:
    # Your CPU code here
    return {"result": "processed"}

@remote(resource_config=config, method="GET", path="/health")
async def health_check() -> dict:
    return {"status": "healthy"}
```

## Custom Docker images

For specialized environments requiring pre-built Docker images—such as vLLM, TensorRT-LLM, or custom ML frameworks—use `ServerlessEndpoint` (GPU) or `CpuServerlessEndpoint` (CPU). These configurations deploy traditional Runpod Serverless endpoints using your own Docker images instead of Flash's managed workers.

See [Custom Docker images](/flash/configuration/custom-docker-images) for configuration options, examples, and usage patterns.
